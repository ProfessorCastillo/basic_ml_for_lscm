# R Package Development Guide for Claude Code

A step-by-step playbook for building teaching-oriented R packages with Claude Code. Based on the process used to build the `wagnerwhitin` and `basicMLforLSCM` packages for BUSML 4382 at Ohio State.

**Author:** Professor Vince Castillo
**Audience:** A Claude Code agent assisting with R package development

---

## Prerequisites

Before starting, ensure the following are available on the machine:

- **R** (version 4+)
- **RStudio** (optional but helpful for manual testing)
- **R packages:** `devtools`, `roxygen2`, `testthat`
- **Git** with push access to GitHub (use `gh auth setup-git` if using GitHub CLI with HTTPS)

---

## Phase 1: Planning

### 1.1 — Write a PRD

Create a `PRD.md` in the repo root. It should define:

- **Purpose** — what problem this package solves and who it's for
- **Function specifications** — name, parameters, return values for every exported function
- **Output format** — exact expected print output, plot descriptions
- **Test cases** — at least 2 worked examples with known expected results
- **Design constraints** — what dependencies are allowed, what to avoid
- **Package structure** — file layout

### 1.2 — Create an implementation plan

Before writing code, produce a detailed plan covering:

- File creation order (what depends on what)
- Algorithm pseudocode for non-trivial functions
- Naming conventions (be consistent across all functions)
- Parameter naming (decide once, apply everywhere — e.g., if two functions share a concept, use the same parameter name in both)
- Test case expected values (compute by hand or verify independently)

**Lesson learned:** Parameter inconsistency between functions causes student confusion. Decide on a shared interface early and stick to it.

## Phase 2: Package Scaffolding

### 2.1 — Directory structure

```
package-name/
├── DESCRIPTION
├── LICENSE
├── NAMESPACE
├── .Rbuildignore
├── .gitignore
├── R/
│   ├── main_function.R
│   ├── helpers.R
│   └── utils.R
├── man/                  # auto-generated by roxygen2
├── tests/
│   ├── testthat.R
│   └── testthat/
│       ├── test_main.R
│       └── test_helpers.R
├── README.md
└── PRD.md
```

### 2.2 — DESCRIPTION

```
Package: packagename
Type: Package
Title: One-Line Title
Version: 0.1.0
Authors@R: person("First", "Last", email = "email@osu.edu",
    role = c("aut", "cre"))
Description: Multi-line description of what the package does.
License: MIT + file LICENSE
Encoding: UTF-8
Imports: graphics, stats
Suggests:
    openxlsx,
    testthat (>= 3.0.0)
Config/testthat/edition: 3
RoxygenNote: 7.3.3
```

**Key points:**
- `RoxygenNote` must match the installed roxygen2 version. Check with `packageVersion("roxygen2")`.
- Use `Suggests` for optional dependencies (like `openxlsx` for Excel export). Use `Imports` for required ones.
- Only `graphics` and `stats` from base R need to be listed in Imports for R CMD check compliance.

### 2.3 — LICENSE

For `License: MIT + file LICENSE` in DESCRIPTION, the LICENSE file must be DCF format:

```
YEAR: 2026
COPYRIGHT HOLDER: Your Name
```

**Not** the full MIT license text. R CMD check will throw a NOTE if this is wrong.

### 2.4 — NAMESPACE

Write a placeholder — roxygen2 will regenerate it:

```
# Generated by roxygen2: do not edit by hand
```

### 2.5 — .Rbuildignore

Exclude non-package files from the build:

```
^\.claude$
^PRD\.md$
^LICENSE\.md$
^.*\.Rproj$
^\.Rproj\.user$
^AI_ASSISTANT_SYSTEM_PROMPT\.md$
^R_PACKAGE_DEV_GUIDE\.md$
```

Add any other `.md` docs, data files, or tooling configs that aren't part of the R package.

### 2.6 — tests/testthat.R

Boilerplate test runner (never changes):

```r
library(testthat)
library(packagename)

test_check("packagename")
```

## Phase 3: Implementation

### 3.1 — Write R source files

**Roxygen2 documentation pattern:**

```r
#' Function Title
#'
#' Description of what the function does.
#'
#' @param x numeric; description of parameter
#' @param y character; description of parameter
#'
#' @return Description of return value
#'
#' @examples
#' my_function(x = 1, y = "a")
#'
#' @export
my_function <- function(x, y) {
  # implementation
}
```

**Rules:**
- Every exported function needs `@export` and a complete roxygen block
- Internal helpers use `@noRd` to suppress documentation generation
- Use `@importFrom` for specific functions from other packages (e.g., `@importFrom stats qnorm sd`)
- For optional dependencies, use `requireNamespace("pkg", quietly = TRUE)` at runtime instead of `@importFrom`

**S3 methods pattern:**

```r
#' @export
print.MyClass <- function(x, ...) {
  # print logic
  invisible(x)  # always return invisible(x) from print methods
}

#' @export
plot.MyClass <- function(x, ...) {
  old_par <- par(no.readonly = TRUE)
  on.exit(par(old_par))  # always save/restore par
  # plot logic
}
```

### 3.2 — Write tests

**Test file structure:**

```r
# ---------- Section Name ----------

test_that("description of what is tested", {
  result <- my_function(inputs)
  expect_equal(result$value, expected_value)
})
```

**What to test:**
1. **Input validation** — bad types, NAs, negatives, wrong lengths
2. **Known examples** — at least 2 worked examples with hand-verified expected values
3. **Edge cases** — single element, zeros, boundary conditions
4. **Return structure** — correct class, correct field names
5. **S3 methods** — print returns `invisible(x)`, plot doesn't error
6. **Optional features** — use `skip_if_not_installed("pkg")` for tests requiring suggested packages

### 3.3 — Excel export (if needed)

A reusable pattern for exporting results to `.xlsx`:

```r
#' @export
export_xlsx <- function(x, file) {
  if (!inherits(x, "MyClass"))
    stop("x must be a MyClass object", call. = FALSE)
  if (!requireNamespace("openxlsx", quietly = TRUE))
    stop("Package 'openxlsx' is required. Install with: install.packages('openxlsx')",
         call. = FALSE)

  wb <- openxlsx::createWorkbook()
  openxlsx::addWorksheet(wb, "Sheet Name")
  openxlsx::writeData(wb, "Sheet Name", my_data_frame)
  openxlsx::saveWorkbook(wb, file, overwrite = TRUE)
  invisible(file)
}
```

## Phase 4: Build and Check

Run these in order after every round of changes:

```r
devtools::document()   # generate man/ pages and NAMESPACE from roxygen2
devtools::test()       # run testthat suite
devtools::check()      # full R CMD check
```

**Targets:**
- `devtools::test()` — all expectations pass, 0 failures
- `devtools::check()` — 0 errors, 0 warnings, 0 notes

**Common issues and fixes:**

| Issue | Fix |
|-------|-----|
| `Undocumented code objects` | Run `devtools::document()` to generate `.Rd` files |
| `License stub is invalid DCF` | Make LICENSE file DCF format (YEAR + COPYRIGHT HOLDER only) |
| `Non-standard file/directory found at top level` | Add the file to `.Rbuildignore` |
| `Hidden files and directories` | Add `.claude` (or similar) to `.Rbuildignore` |
| `RoxygenNote version mismatch` | Update `RoxygenNote` in DESCRIPTION to match installed version |

## Phase 5: Documentation

### 5.1 — README.md

Structure for a teaching-oriented package:

1. **Title and one-line description**
2. **Installation** — `devtools::install_github()` instructions
3. **Quick Start** — full worked example with code and expected output
4. **Understanding the Output** — section-by-section explanation of what the functions return
5. **How It Works** — conceptual explanation of the algorithm/method (not just API docs)
6. **Parameter Notes** — clarify any unit conventions, annualization, etc.
7. **Function Reference** — table of all exported functions with one-line descriptions
8. **Sample Code** — multiple examples covering all exported functions
9. **AI Assistant** — link to the system prompt if one exists

### 5.2 — AI Assistant System Prompt

Create an `AI_ASSISTANT_SYSTEM_PROMPT.md` to help students use the package. Structure:

1. **Identity and tone** — who the assistant is, how it behaves
2. **Step-by-step onboarding flow** — check RStudio → install devtools → install package → load → run example → explain output
3. **Output reference** — exact expected output so the assistant can verify student results
4. **Behavior rules** — one step at a time, wait for confirmation, never skip errors
5. **Troubleshooting** — common errors and how to resolve them

Add to `.Rbuildignore` so it doesn't ship with the package.

## Phase 6: Git Workflow

### Commit pattern

Commit after each logical unit of work, not in giant batches:

```bash
git add <specific files>
git commit -m "Short description of change"
git push origin main
```

### Push setup (if using GitHub CLI)

If `git push` fails with credential errors:

```bash
gh auth setup-git          # configure gh as git credential helper
git push origin main       # now works via HTTPS
```

Or switch to SSH if keys are configured:

```bash
git remote set-url origin git@github.com:User/repo.git
```

## Checklist: Before Shipping

- [ ] All `devtools::test()` expectations pass
- [ ] `devtools::check()` returns 0 errors, 0 warnings, 0 notes
- [ ] README has installation instructions, quick start, and all function examples
- [ ] Every exported function has roxygen2 documentation with `@examples`
- [ ] Parameter names are consistent across all functions
- [ ] `.Rbuildignore` excludes all non-package files (PRD, guides, .claude, etc.)
- [ ] Manual test: install from GitHub with `devtools::install_github()`, restart R, run examples
- [ ] AI assistant system prompt is accurate and reflects final function signatures

## Iteration Lessons from wagnerwhitin

Things that came up during development that are worth knowing ahead of time:

1. **Decide parameter names once.** If multiple functions share a concept (e.g., a cost rate, a threshold, a time horizon), use the same parameter name in all of them. Refactoring parameter names after tests and docs are written is tedious and error-prone.

2. **Unit consistency matters for teaching.** If some functions need inputs in different units than others (e.g., monthly vs. annual), handle the conversion inside the function and document it clearly. Don't make students do the conversion themselves.

3. **Don't fight base R plotting.** Legend placement in single-panel plots was a recurring issue. Side-by-side panels (`par(mfrow = c(1, 2))`) with separate subtitles was simpler and cleaner than trying to overlay a legend.

4. **Excel export is high-value for students.** Adding `export_xlsx()` with `openxlsx` as a suggested dependency was straightforward and gives students a deliverable they can submit or paste into reports.

5. **LICENSE file format is picky.** `MIT + file LICENSE` in DESCRIPTION requires the LICENSE file to be DCF format (just YEAR and COPYRIGHT HOLDER), not the full MIT text.

6. **Always run `devtools::document()` before `devtools::check()`.** Missing `.Rd` files are the most common check warning.

7. **Test expected values by hand first.** Compute at least one example manually to verify the algorithm before writing tests. Tests that encode wrong expected values are worse than no tests.

## Iteration Lessons from basicMLforLSCM

Additional lessons from building the interactive ML regression package:

8. **R package names cannot contain underscores.** Only letters, numbers, and dots are allowed. Use camelCase (e.g., `basicMLforLSCM`) for readability. The GitHub repo name can still use underscores (`basic_ml_for_lscm`) — they're independent.

9. **Interactive `readline()` functions can't be tested headlessly.** Design every interactive function with an `interactive = TRUE/FALSE` flag. When `FALSE`, it accepts parameters directly and skips all `readline()` calls. Tests always call with `interactive = FALSE`. This also enables a single-shot non-interactive mode (`ml_run()`) for free.

10. **Students will make input mistakes — design for forgiveness.** Beginners enter column numbers (`[3]`) instead of names, use wrong case (`stockouts` instead of `Stockouts`), and accidentally answer "yes" to the wrong prompt. Build input resolution that accepts numbers, bracket-wrapped numbers, and case-insensitive names. Provide escape hatches (empty input to skip, confirmation prompts to redo).

11. **Go-back navigation is essential for interactive workflows.** Students will advance past a step by mistake. Every step should offer a way to return to the previous step without starting over. Use a state machine in the orchestrating function — each step returns `list(go_back = TRUE)` to signal backward navigation, and the orchestrator decrements to the previous step.

12. **State passing via lists, not global variables.** Chain step functions by having each return a list that the next step unpacks. Each result is a superset of the previous (carries all prior fields plus new ones). This keeps state explicit and testable.

13. **`car::vif()` output format changes with categorical predictors.** It returns a named numeric vector for continuous-only models, but a matrix with GVIF/Df/GVIF^(1/(2*Df)) columns when factors are present. Check `is.matrix()` to detect which format and display accordingly.

14. **`ggplot2` plots don't render inside functions without `print()`.** When calling `ggplot()` inside a function, you must explicitly `print(gg)` for the plot to appear. Base R `plot()` renders automatically. The `plot.ml_result()` S3 method uses base R for 2-panel layouts to avoid this issue.

15. **`.Rbuildignore` vs `.gitignore` serve different purposes.** `.Rbuildignore` excludes files from the R package tarball but keeps them in the repo. `.gitignore` excludes files from git entirely. Planning docs (PRD, implementation plan, AI prompts) go in `.Rbuildignore` so they're pushed to GitHub but don't ship with `install.packages()`.

16. **Don't list unused packages in Imports.** R CMD check flags any package in the `Imports` field of DESCRIPTION that isn't actually imported via `@importFrom` or `::`. If you remove a dependency from the code, remove it from DESCRIPTION too. During the basicMLforLSCM build, `dplyr` and `utils` were initially listed but never used, causing NOTEs.

17. **GitHub repo initialization conflicts.** If the GitHub repo was created with a README or LICENSE via the web UI, `git push` will be rejected on the first push. Use `git pull --rebase --allow-unrelated-histories` to integrate the remote content before pushing. Resolve merge conflicts by keeping your package's versions (especially the DCF-format LICENSE).

18. **Test data files must be included in the package build.** If tests reference xlsx files in `tests/testthat/testdata/`, do NOT add that directory to `.Rbuildignore`. `devtools::test()` runs from the source directory (finds the files), but `R CMD check` builds a tarball first — excluded files won't be in the check environment, and all tests that need them will fail.

19. **Keep the AI assistant prompt in sync with every code change.** When adding features like go-back navigation or flexible input, update the AI assistant system prompt in the same commit or immediately after. Students will ask the AI for help, and stale guidance (e.g., "you'll see 6 options" when there are now 7) causes confusion.

20. **Manual testing with real students catches things unit tests don't.** A student typing `[3]` instead of a column name, or accidentally hitting `y` at the wrong prompt, revealed UX issues that no automated test would surface. Always do a manual walkthrough from the student's perspective before shipping.
